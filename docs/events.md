## Событийная модель Payment Gateway

Документ описывает доменные события, которые публикует платёжный шлюз в Kafka. События используются для построения аналитики, интеграции с внешними системами (CRM, биллинг, отчётность, риск‑менеджмент) и расширения функциональности, не влияя на основной поток обработки платежей.

---

## Общие принципы

- **Транспорт**: Kafka (допускается аналогичный брокер сообщений).
- **Формат сообщений**: JSON.
- **Ключ сообщения**: как правило, идентификатор платежа (`paymentId`) или возврата (`refundId`), чтобы обеспечить упорядоченность по ключу.
- **Семантика доставки**: как минимум once (подписчики должны быть идемпотентны).
- **Источник истины**: база данных платёжного шлюза; события отражают изменения состояния доменных сущностей.

---

## Обзор событий

В рамках MVP используются следующие события:

- **`payment_created`** — платёж создан;
- **`payment_succeeded`** — платёж успешно завершён;
- **`payment_failed`** — платёж завершился неуспехом;
- **`refund_requested`** — запрошен возврат по платежу;
- **`refund_completed`** — возврат успешно завершён.

Каждое событие описывает изменение состояния сущности (платежа или возврата) и может содержать как текущее состояние, так и минимальный набор полей для идентификации.

---

## Общая структура событий

Рекомендуется единый "оболочный" формат для всех доменных событий:

- **`eventType`** (`string`) — тип события (например, `payment_created`);
- **`eventId`** (`string`) — уникальный идентификатор события;
- **`occurredAt`** (`string`, ISO 8601) — момент времени, когда событие произошло в домене;
- **`payload`** (`object`) — полезная нагрузка, зависящая от типа события.

Пример общей структуры:

```json
{
  "eventType": "payment_created",
  "eventId": "evt_12345",
  "occurredAt": "2026-03-10T10:15:30Z",
  "payload": {
    "...": "зависит от конкретного события"
  }
}
```

Ниже описаны поля `payload` для каждого события.

---

## Событие `payment_created`

**Назначение**: зафиксировать факт создания нового платежа.

Рекомендуемый топик: `payments.lifecycle` (или аналогичный, в зависимости от принятой конвенции именования).

Полезная нагрузка (`payload`):

- **`paymentId`** (`string`) — идентификатор платежа;
- **`orderId`** (`string`) — идентификатор заказа в системе магазина;
- **`amount`** (`number`) — сумма;
- **`currency`** (`string`) — валюта (MVP: `EUR`);
- **`status`** (`string`) — на момент события обычно `PENDING`;
- **`customerId`** (`string`, опционально) — идентификатор клиента, если есть;
- **`redirectUrl`** (`string`, опционально) — URL для переадресации, если используется redirect‑флоу.

Использование:

- аналитика по вновь созданным платежам;
- синхронизация с внешними системами заказов/биллинга.

---

## Событие `payment_succeeded`

**Назначение**: уведомить о том, что платёж успешно проведён внешним провайдером.

Рекомендуемый топик: тот же, что и для `payment_created` (например, `payments.lifecycle`).

Полезная нагрузка (`payload`):

- **`paymentId`** (`string`);
- **`orderId`** (`string`);
- **`amount`** (`number`);
- **`currency`** (`string`);
- **`status`** (`string`) — значение `SUCCEEDED`;
- **`providerTransactionId`** (`string`, опционально) — идентификатор операции у внешнего провайдера;
- **`completedAt`** (`string`, ISO 8601, опционально) — момент успешного завершения.

Использование:

- перевод соответствующего заказа в статус "оплачен" во внешних системах;
- финансовая и управленческая отчётность;
- триггер для отгрузки товара/оказания услуги.

---

## Событие `payment_failed`

**Назначение**: зафиксировать неуспешную попытку оплаты.

Рекомендуемый топик: `payments.lifecycle` или отдельный, например `payments.failures`.

Полезная нагрузка (`payload`):

- **`paymentId`** (`string`);
- **`orderId`** (`string`);
- **`amount`** (`number`);
- **`currency`** (`string`);
- **`status`** (`string`) — значение `FAILED`;
- **`failureCode`** (`string`) — один из кодов ошибок (например, `INVALID_CARD`, `INSUFFICIENT_FUNDS`, `CARD_DECLINED`, `EXPIRED_CARD`, `PAYMENT_EXPIRED`);
- **`failureMessage`** (`string`) — человекочитаемое описание причины;
- **`failedAt`** (`string`, ISO 8601, опционально).

Использование:

- анализ отказов по платёжным провайдерам, странам, банкам;
- антифрод и риск‑менеджмент;
- улучшение UX (например, рекомендации по способам оплаты).

---

## Событие `refund_requested`

**Назначение**: отразить запрос на возврат средств по уже проведённому платежу.

Рекомендуемый топик: `payments.refunds` (или иной, принятый для операций возвратов).

Полезная нагрузка (`payload`):

- **`refundId`** (`string`) — идентификатор возврата;
- **`paymentId`** (`string`) — связанный платёж;
- **`orderId`** (`string`);
- **`amount`** (`number`) — сумма запрошенного возврата;
- **`currency`** (`string`);
- **`status`** (`string`) — значение `REQUESTED`.

Использование:

- фиксация обязательств по возвратам;
- запуск внутренних процессов (например, уведомление бухгалтерии или службы поддержки).

---

## Событие `refund_completed`

**Назначение**: уведомить о фактическом завершении возврата средств.

Рекомендуемый топик: тот же, что и для `refund_requested` (`payments.refunds`).

Полезная нагрузка (`payload`):

- **`refundId`** (`string`);
- **`paymentId`** (`string`);
- **`orderId`** (`string`);
- **`amount`** (`number`);
- **`currency`** (`string`);
- **`status`** (`string`) — значение `REFUNDED`;
- **`providerRefundId`** (`string`, опционально) — идентификатор возврата у провайдера;
- **`completedAt`** (`string`, ISO 8601, опционально).

Использование:

- обновление финансовых регистров и аналитики;
- информирование клиента о завершении возврата;
- контроль SLA по обработке возвратов.

---

## Замечания по проектированию подписчиков

- **Идемпотентность**: подписчики должны корректно обрабатывать повторную доставку одного и того же события (по `eventId` или по доменному идентификатору).
- **Расширяемость**: возможно добавление новых полей в `payload`; подписчики не должны ломаться при появлении неизвестных полей.
- **Трассировка**: рекомендуется передавать корреляционные идентификаторы (например, `traceId`, `correlationId`) для сквозной трассировки цепочки вызовов.

---

## Итоги

Событийная модель Payment Gateway позволяет строить вокруг платёжного ядра дополнительные сервисы (аналитика, отчётность, антифрод, уведомления) без жёсткой синхронной связи. В дальнейшем список событий может быть расширен (чарджбэки, pre‑authorization, recurring‑платежи и т.д.), сохраняя при этом общие принципы структуры и семантики.
