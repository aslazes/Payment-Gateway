## Обзор архитектуры

Payment Gateway — это внутренний платёжный шлюз для группы интернет‑магазинов. Он предоставляет единый REST API для приёма платежей банковскими картами через внешнего платёжного провайдера (банк‑эквайер), а также публикует события о платежах в Kafka для последующей аналитики и интеграций.

Основные цели архитектуры:

- **Единая точка интеграции** для всех интернет‑магазинов (merchant‑систем).
- **Инкапсуляция интеграции** с внешним платёжным провайдером.
- **Надёжный учёт платежей и возвратов** во внутренней базе данных.
- **Событийная интеграция** через Kafka для построения аналитики и дополнительных сервисов.

## Высокоуровневая схема

На концептуальном уровне система выглядит так:

- **Merchant System (интернет‑магазин)**: инициирует платежи, запрашивает статусы, выполняет возвраты, получает историю платежей.
- **Payment Gateway API**: единая точка входа в виде REST API.
- **Domain‑сервисы** внутри шлюза:
  - `Payment Service` — создание платежей и взаимодействие с внешним провайдером.
  - `Payment Status Service` — получение и обновление статусов платежей.
  - `Refund Service` — обработка полных и частичных возвратов.
- **External Payment Provider (Acquirer)**: внешний банк‑эквайер/провайдер, который фактически списывает и возвращает средства.
- **Database**: хранилище транзакций, статусов и связанной служебной информации.
- **Kafka**: брокер сообщений, через который публикуются события об изменении состояния платежей.

## Основные компоненты

### 1. REST API

Интерфейс системы описан спецификацией OpenAPI (`api/openapi.yaml`) и включает следующие ключевые ресурсы:

- **`POST /payments`** — создание платежа на основе `PaymentCreate`.
- **`GET /payments`** — получение списка платежей по `orderId` и/или `customerId`.
- **`GET /payments/{paymentId}`** — получение детального статуса платежа.
- **`POST /payments/{paymentId}/refunds`** — создание возврата по существующему платежу.

API спроектирован так, чтобы:

- минимизировать объём обязательных данных (orderId, amount, currency, cardToken);
- явно возвращать бизнес‑ошибки в виде `ErrorResponse` с кодами (например, `INVALID_CARD`, `INSUFFICIENT_FUNDS`, `CARD_DECLINED`, `REFUND_NOT_ALLOWED` и др.);
- быть устойчивым к эволюции (расширяемые схемы, использование enum‑значений, nullable‑поля).

### 2. Сервис платежей (`Payment Service`)

Отвечает за полный жизненный цикл платежа:

- приём запроса от API (`PaymentCreate`);
- валидацию бизнес‑ограничений (валюта, дубликат заказа и т.п.);
- создание записи о платеже во внутренней БД со статусом `PENDING`;
- подготовку и отправку запроса во внешний провайдер (redirect URL или токенизированные данные карты);
- обработку ответа провайдера и перевод статуса в `SUCCEEDED` или `FAILED`;
- публикацию соответствующих событий в Kafka (`payment_created`, `payment_succeeded`, `payment_failed`).

Сервис не хранит "сырые" данные банковских карт — используется карточный токен (`cardToken`), полученный от безопасного хранилища/платёжного провайдера.

### 3. Сервис статусов (`Payment Status Service`)

Реализует чтение текущего состояния платежа:

- обслуживает запрос `GET /payments/{paymentId}`;
- при необходимости может инициировать фоновую синхронизацию статуса с внешним провайдером;
- возвращает клиенту агрегированную информацию по платежу (amount, currency, status, redirectUrl).

Также сервис может использоваться внутренними компонентами для проверки возможности возврата или повторной попытки платежа.

### 4. Сервис возвратов (`Refund Service`)

Отвечает за обработку возвратов:

- принимает запросы на возврат по конкретному платежу (`POST /payments/{paymentId}/refunds`);
- проверяет допустимость операции (платёж существует, успешен, не просрочен, сумма возврата не превышает оплаченной и т.д.);
- создаёт запись возврата в БД со статусом `REQUESTED`;
- инициирует операцию возврата во внешнем провайдере;
- по результату переводит статус возврата в `REFUNDED` или `FAILED`;
- публикует события `refund_requested` и `refund_completed` в Kafka.

Поддерживаются как полные, так и частичные возвраты (сумма указывается явно в `RefundCreate.amount`).

### 5. База данных

В рамках MVP предполагается единая реляционная база данных, в которой хранятся:

- платежи (paymentId, orderId, amount, currency, status, технические метаданные);
- возвраты и связь с платежами;
- вспомогательные сущности (например, история запросов к внешнему провайдеру).

База данных является "источником истины" для внутренних сервисов, а Kafka — каналом распространения информации наружу.

### 6. Kafka и событийная модель

Сервис публикует доменные события при изменении состояния платежей и возвратов:

- `payment_created`
- `payment_succeeded`
- `payment_failed`
- `refund_requested`
- `refund_completed`

События сериализуются в формате JSON и попадают в соответствующие топики Kafka. Другие системы (аналитика, биллинг, риск‑менеджмент и т.п.) могут подписываться на эти события и строить поверх них свои процессы, оставаясь слабо связаными с самим платёжным шлюзом.

## Обработка основных сценариев

### Успешная оплата

1. Интернет‑магазин вызывает `POST /payments` с параметрами заказа.
2. Payment Gateway создаёт платёж со статусом `PENDING`, запрашивает проведение операции во внешнем провайдере.
3. В зависимости от выбранной схемы интеграции:
   - клиент перенаправляется на страницу провайдера (`redirectUrl`),
   - либо используется ранее полученный `cardToken`.
4. После авторизации/подтверждения провайдер возвращает результат.
5. Статус во внутренней системе обновляется до `SUCCEEDED`, публикуется событие `payment_succeeded`.
6. Интернет‑магазин может периодически опрашивать `GET /payments/{paymentId}` или использовать события для обновления статуса заказа.

### Неуспешная оплата

1. Процесс аналогичен успешной оплате, но провайдер возвращает отказ (например, `CARD_DECLINED`, `INSUFFICIENT_FUNDS`, `EXPIRED_CARD`).
2. Статус платежа меняется на `FAILED`, публикуется событие `payment_failed`.
3. Клиенту возвращается соответствующая ошибка (`ErrorResponse`) с кодом и человекочитаемым сообщением.

### Запрос статуса платежа

1. Интернет‑магазин вызывает `GET /payments/{paymentId}`.
2. `Payment Status Service` читает данные из БД и при необходимости синхронизируется с внешним провайдером.
3. Клиент получает текущее состояние (`PENDING`, `SUCCEEDED`, `FAILED`) и дополнительную информацию (например, `redirectUrl`).

### Возврат средств

1. Интернет‑магазин инициирует возврат по успешному платежу через `POST /payments/{paymentId}/refunds`.
2. `Refund Service` проверяет бизнес‑правила и отправляет запрос во внешний провайдер.
3. При успешном завершении операция фиксируется в БД со статусом `REFUNDED`, публикуется `refund_completed`.
4. В случае ошибки статус меняется на `FAILED`, клиент получает `ErrorResponse` с соответствующим кодом.

## Ограничения и принятые упрощения MVP

Для демонстрации архитектуры приняты следующие допущения:

- поддерживается только **один** платёжный провайдер;
- используется одна валюта — **EUR**;
- обрабатываются только **платежи по банковским картам**;
- данные карт не хранятся в системе (используются токены/redirect‑флоу);
- нет сложной маршрутизации, антифрода, мультивалютности и др.

Эти ограничения упрощают архитектуру и позволяют сосредоточиться на ключевых аспектах: контракте API, событиях и базовой оркестрации бизнес‑процессов.

## Возможные направления развития архитектуры

Архитектура спроектирована так, чтобы систему можно было постепенно усложнять:

- добавить поддержку нескольких провайдеров и маршрутизацию платежей (по стране, банку, типу карты и т.д.);
- вынести антифрод‑логику в отдельный сервис;
- расширить событийную модель (события по чарджбэкам, pre‑authorization, recurring‑платежам и т.п.);
- реализовать подписки/рекуррентные списания;
- внедрить мультивалютность и динамическое управление курсами;
- добавить webhook‑уведомления для интернет‑магазинов поверх Kafka‑событий.
