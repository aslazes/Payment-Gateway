# Требования к системе Payment Gateway MVP

## 1. Контекст и цель

**Система:** внутренний платёжный шлюз для группы интернет-магазинов.  

**Проблема:** сейчас каждый магазин интегрируется с платёжными провайдерами отдельно, что усложняет поддержку и аналитику.  

**Цель MVP:**

- Единый REST API для проведения онлайн-платежей и возвратов  
- Централизованная история транзакций и статусов  
- Событийная архитектура (Kafka) для будущей аналитики, уведомлений и антифрода  
- Поддержка платежей картой и возвратов (MVP)

---

## 2. Внешние акторы

| Актор | Роль |
|-------|------|
| **Интернет-магазин (Merchant System)** | Создаёт платежи, запрашивает статусы, инициирует возвраты через API |
| **Покупатель (Customer)** | Совершает оплату заказа в магазине, напрямую с API не взаимодействует |
| **Внешний платёжный провайдер / банк-эквайер (Acquirer)** | Обрабатывает оплату и возвращает успешный или отклонённый результат |

---

## 3. Основные бизнес-сценарии (MVP)

### 3.1. Успешная оплата

1. Покупатель оформляет заказ.  
2. Магазин вызывает `POST /payments`.  
3. Payment Gateway регистрирует платёж и при необходимости перенаправляет на 3-D Secure.  
4. Получает результат от провайдера.  
5. Возвращает итоговый статус магазину.

### 3.2. Неуспешная оплата

- Провайдер возвращает отказ (недостаточно средств, неверные данные, отказ банка)  
- Статус `FAILED` с кодом причины (`INVALID_CARD`, `INSUFFICIENT_FUNDS`, `CARD_DECLINED`, …)

### 3.3. Просмотр статуса платежа

- Магазин запрашивает статус по `paymentId` или `orderId` (`GET /payments/{paymentId}`)

### 3.4. Возврат средств

1. Магазин инициирует возврат (`POST /payments/{paymentId}/refunds`)  
2. Gateway регистрирует возврат и отправляет запрос провайдеру  
3. Статус возврата: `REQUESTED`, `REFUNDED`, `FAILED`

### 3.5. Получение истории платежей

- Список платежей по `orderId` или `customerId` (`GET /payments`), без аналитики

---

## 4. Нефункциональные требования

- Простое API, минимальные сущности, понятные статусы  
- Время ответа: создание платежа до 5–10 секунд  
- Логирование операций и изменений статусов  
- Событийная архитектура: каждое значимое изменение публикуется в Kafka

---

## 5. Ограничения MVP

- Один внешний провайдер  
- Только платежи картой; кошельки, подписки, холды — позже  
- Валюта — EUR; мультивалютность позже  
- PCI DSS: карты не хранятся, используются токены / redirect

---

## 6. Основные термины

| Термин | Определение |
|--------|-------------|
| **Payment** | Платёж, инициированный магазином |
| **Refund** | Возврат средств по успешному платежу |
| **Payment Gateway** | Внутренний сервис для унифицированной работы с платежами |
| **Status** | Текущий статус платежа или возврата |
| **Event** | Сообщение в Kafka о значимом изменении платежа или возврата |
